# 데일리루프 P2 — 마스코트 결속 설계 (2026-07-23)

데일리루프 P1(플랜·오늘 목표·오늘 홈)이 구현된 위에, **마스코트를 살아있게** 만든다.
마스코트 무드가 오늘치 달성 여부에서 파생되어 홈에서 감정 피드백을 주고,
저녁 알림은 "Shaun이 배고파해요"로 상기시키며, 목표 달성 시 축하 연출이 터진다.

- **대상**: `verse-flutter/` 앱 전용. 오프라인 우선(로컬 파생 계산).
- **선행**: P1 완료(`memorization_plan`, `PlanView.todayDone`, `TodayScreen`).
- **새 테이블 없음** — 기존 `StreakState` + `PlanView`에서 전부 파생한다.
- **마스코트 이름: `Shaun`** (양 스프라이트). 사용자 노출 문구에서 "마스코트"라는 일반명사 대신
  이름을 부른다 — 애착이 생기고 알림·홈 문구가 사람처럼 읽힌다. ko/en 동일한 고유명사로 쓴다.

---

## 1. 마스코트 무드 파생 규칙

기존 `MascotSprite`의 3무드(idle/happy/sad)를 그대로 재사용한다(신규 아트 없음).
순수 함수로 계산해 단위 테스트로 전 케이스를 검증한다.

| 조건 | 무드 |
|---|---|
| 오늘치 완료(`PlanView.todayDone`) | **happy** |
| 오늘 활동은 했으나 목표 미달(`streak.lastDay == 오늘`) | **idle** |
| 오늘 아직 안 함 + 과거 활동 기록 있음(`streak.lastDay < 오늘`) | **sad**(배고픔) |
| 활동 기록 없음(신규) 또는 활성 플랜 없음 | **idle** |

- **듀오링고식**: 매일 아침 앱을 열면 sad(배고픔)로 넛지하고, 오늘치를 채우면 happy로 전환된다.
- "오늘 시작은 했지만 목표 미달"을 sad가 아닌 idle로 두는 이유: 이미 오늘 참여한 사용자에게
  배고픔을 띄우는 것은 과하다. 진행 중이라는 중립 신호가 맞다.
- 날짜 기준은 **UTC 자정**(스트릭·P1 오늘치 판정과 동일).
- 톤: "굶어 죽음"이 아니라 "시무룩/배고픔"까지만. 복귀하면 즉시 회복된다.

---

## 2. 오늘 홈 배치 — 마스코트와 목표를 한 덩어리로

마스코트를 화면의 주인공으로 올리고, **오늘 목표를 바로 아래 붙여 한 블록**으로 묶는다.
목표 바가 차오르면 바로 위 마스코트가 sad → happy로 바뀌어 **인과가 눈에 보인다**.

```
        [ 마스코트 96px ]        ← 무드 = 오늘 목표 달성 여부
         "Shaun이 배고파해요... 오늘 한 절 어때요?"   ← 무드별 한 줄
         🔥 7          ♥ 5       ← 스트릭 · 목숨 (작게)

         오늘 목표
       ▓▓▓▓▓░░░░░  2/3절
```

그 아래는 P1 그대로: 플랜 전체 진행바 → "이어서 외우기" CTA.

**무드별 한 줄 문구**(l10n ko/en, 키 3개):
- sad: "Shaun이 배고파해요... 오늘 한 절 어때요?" / "Shaun is hungry... how about one verse today?"
- idle: "좀만 더 하면 돼요!" / "Almost there — keep going!"
- happy: "오늘도 해냈어요! Shaun도 신났어요" / "You did it today! Shaun is thrilled"

## 3. 완료 축하 연출

**오늘 홈 복귀 시** 터뜨린다. 암송을 끝내고 홈에 돌아왔을 때 `todayDone`이 참이면
마스코트가 happy로 팝하며 컨페티 + 격려 문구가 1회 재생된다.

- **암송 결과 화면이 아닌 홈에서** 하는 이유: `MemorizeController`가 플랜 목표를 알 필요가 없어
  **암송↔플랜 결합이 생기지 않는다**. P1 아키텍처(홈이 파생 계산을 소유)와 일치한다.
- **하루 1회 가드**: 마지막 축하일(`YYYY-MM-DD`)을 기존 `SyncMeta` 키-값 테이블에 저장한다.
  오늘 이미 축하했으면 재생하지 않는다 → **새 테이블 불필요**.
- 연출은 기존 자산 재사용(섹션 완료 화면의 happy 마스코트, `pixel_burst`의 버스트 효과).

## 4. 배고픔 알림 문구 교체

기존 스트릭 위험 알림(저녁, ID 1002)은 이미 "어제까지 스트릭이 살아있고 오늘 미완료"일 때만
발사된다 — **§1의 sad 조건과 정확히 일치**한다. 따라서 **스케줄링 로직은 손대지 않고 문구만 교체**한다.

- 제목: `"스트릭이 위험해요!"` → **`"Shaun이 배고파해요 🍞"`**
- 본문: 스트릭 언급을 **유지**한다. Shaun만 말하면 실제로 잃는 것(스트릭)이 흐려지므로,
  둘을 함께 걸어 감정 훅과 손실 회피가 같이 작동하게 한다.
  예: `"한 절로 Shaun을 먹여주세요 — N일 스트릭도 오늘 밤 끊겨요!"`
- 알림은 `BuildContext` 없이 예약되므로 이름은 l10n이 아니라 `mascotName` 상수를 쓴다.
- 기존 3종 문구 로테이션 구조와 ko/en 분기는 그대로 유지한다.
- 매일 리마인더(ID 1001)와 일시정지 안내(ID 1003)는 **변경하지 않는다**.

## 5. 데이터 & 재사용

**신규 테이블: 없음.**

| 필요한 것 | 출처 |
|---|---|
| 오늘치 완료 여부 | `PlanView.todayDone` (P1) |
| 마지막 활동일 | `StreakState.lastDay` (기존) |
| 마스코트 스프라이트 | `MascotSprite`(idle/happy/sad, 기존 SVG 4장) |
| 축하 1회 가드 | `SyncMeta` 키-값 (기존) |
| 알림 스케줄링 | `ReminderService.refreshStreakDanger` (기존, 문구만 교체) |

---

## 6. 테스트

- **무드 파생 순수 함수**: §1 표의 4개 분기 + 경계(어제 활동/2일 공백/플랜 없음) 단위 테스트.
- **홈 위젯**: 무드별로 해당 스프라이트와 문구가 렌더되는지.
- **축하 1회 가드**: 같은 날 두 번 홈에 들어와도 축하가 한 번만 재생되는지.
- **알림 문구**: 교체된 제목/본문이 ko/en으로 나오는지(기존 `streak_danger_test` 확장).

---

## 7. 범위 밖

- 배고픔 **단계 세분화**(중간 단계 아트) — 3무드로 충분하다고 판단, 향후 필요 시.
- **앱 아이콘 동적 변경** — iOS는 변경 시마다 시스템 팝업이 강제되고 Android는 사실상 불가.
  자동 변경은 채택하지 않는다. (수동 아이콘 선택 기능은 별개 주제.)
- 먹이주기 등 **다마고치 인터랙션** — 데일리루프의 부산물로 두고 별도 기능화하지 않는다.
- 카드 컬렉션 — 별개 스펙.
