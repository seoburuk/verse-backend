# 픽셀 디자인 개선 계획

작성일: 2026-07-19 (2026-07-19 컨셉 확장 반영)
배경: [Shepherd 앱 분석](competitor-shepherd-app.md), [성경 암송 앱 UX 불만 조사](competitor-bible-memorization-apps.md), 현재 [globals.css](../verse-web-next/app/globals.css) 검토 결과

## 컨셉: 포켓몬 골드풍 게임기 UI

전체 방향은 **"게임보이/포켓몬 골드 시대의 휴대용 게임기"** 감성. 정체성 3요소:
1. 두꺼운 테두리의 창(window) 단위 UI — 텍스트가 맨바닥에 뜨지 않고 항상 박스 안에
2. 화면 밖으로 나가지 않는 정돈된 프레임 (게임기 베젤)
3. 제한된 팔레트 (녹흑 게임보이 LCD 계열)

Shepherd류 둥글둥글 파스텔로 **전환하지 않는다**. 성경 앱에서 픽셀은 희소해 차별화가 공짜로 되고, 게임화 시스템(목숨·XP·스테이지)과 픽셀 미학이 문법적으로 맞는다. Shepherd에서는 **문법**(펫의 감정 상태, 데일리 루프, 통통 튀는 모션)만 빌린다.

현재 구조(하드섀도, `steps()` 스텝 애니메이션, 콤보/컨페티/파티클)는 견고. 문제는 구조가 아니라 **온도**(팔레트가 터미널 같음)와 **밀도**(그래픽 자산 0개).

## 핵심 결정 사항 (2026-07-19 논의)

### A. 게임화 컨셉: 픽셀 동물 카드 수집 (포켓몬 카드 팩 문법)

**결정 배경**: 코어 가치를 "성경 암송을 **가볍게** 게임처럼"으로 고정한 뒤 후보(다마고치 / 듀오링고식 / 양떼 / 카드 수집)를 재평가한 결과, **다마고치는 코어와 방향 충돌**(밥주기·진화·알림은 "가볍게"의 반대인 매일 돌봄 의무). 최종 채택은 **동물 카드 수집**.

- **채택 이유**: "가볍게"(벌·의무 없음, 외우면 카드 하나 뜰 뿐) + "게임처럼"(팩 개봉·희귀도·도감) + 저비용(카드 로스터가 유한) + 차별화(성경 앱에 카드 수집 없음)를 모두 만족.
- **폐기한 것**: 다마고치의 밥주기·진화 게이지·강제 알림. 애착·수집 재미는 카드가 그대로 흡수하고 의무감만 제거.
- **핵심 구조**: 카드 **종류는 고정 로스터(30~50종)**, 획득 **계기는 무한**(구절/섹터 클리어). 이게 "3만 절을 카드화하면 압도적" + "구절마다 일러스트 못 그림" 두 문제를 동시에 해결.
- **로스터 구성**: 성경 동물(양·비둘기·까마귀·독수리·수탉·사자·뱀 등)은 성경 근거로 스토리 부여 + 일반/재미 동물(코끼리·공룡 등)은 순수 수집 재미·희귀 카드로. 성경+엉뚱함의 조합이 "가볍게"에 정합.
- **기존 자산 재활용**: 이미 만든 비둘기 스프라이트(idle/happy/sad, GBC 삐죠 톤)를 첫 카드 일러스트 및 마스코트로 그대로 사용. 다크모드 대비도 확인됨.

### B. 명시적 데일리 루프: "하루 3승"의 암송 버전
홈 최상단에 **퀘스트 보드**(체크박스 박스 UI):
1. `[ ] 오늘의 새 구절` — 새 구절 1개 암송 (3~5분)
2. `[ ] 복습 퀴즈` — 기존 구절 중 N개(2~3개) 리콜 테스트 (나중에 SRS 간격반복으로 발전 가능한 자리)
3. 둘 다 완료 → 펫 기뻐함 + 스트릭 +1 + XP 획득
- 총 소요 **5분 이내** 목표 (경쟁 앱 조사: "짧은 일상 루틴화" 선호 확인).
- 현재 `resume-card` 자리를 이 데일리 보드로 교체.

### C. 포지셔닝: "성경 암송의 듀오링고"
- **랜딩·SNS·보도자료에서만** 사용 (스타트업 관용구, 관행상 문제 소지 낮음).
- **스토어 메타데이터(앱 이름·부제·키워드)에는 넣지 않음** — 타사 상표 언급은 Apple/Google 심사 거절 사유 가능, Duolingo는 상표 방어에 적극적.
- 스토어용 자체 문구: "성경 암송을 게임처럼" 류.
- 상표 이슈 걱정되면 전문가 확인 권장 (본 문서는 법률 자문 아님).

### D. 팔레트: 검+초록 유지, 순검정만 탈피
게임보이 LCD 정체성이라 컨셉과 정합. "터미널 느낌"은 순검정(#0f0f0f)+쨍한 초록 탓 → 초록을 버리지 말고 **배경을 초록 쪽으로 끌어옴**:
- 다크 배경: `#0f0f0f` → `#0e1410`(아주 어두운 녹흑색)
- 다크 서피스: `#1a1a1a` → `#16211a`
- 액센트 초록: `#4ade80` → 채도 낮춘 `#5cbf82` 계열
- 결과: "해커 터미널"이 아니라 "초록빛 도는 게임보이 화면"
- 라이트 모드도 동일 기조, 크림/양피지 톤 유지, 대비 접근성 재확인

### E. 레이아웃: 좌우 여백 유지 (풀블리드 금지)
- 하드섀도(4px 4px 0)는 오른쪽·아래 공간이 있어야 보임 → 풀블리드로 붙이면 그림자가 잘려 픽셀 정체성이 깨짐.
- 현재처럼 좌우 1rem 여백 + 보더 박스 유지, 눌림 촉감(`translate(4px,4px)`) 유지.
- 화면을 채우고 싶으면 버튼을 늘리지 말고 **게임기 베젤 프레임(`.page` 보더)을 모바일에서도 살림**.

### F. Flutter 모션 구현 방침 (참고 — 백엔드 개발자용)
픽셀 모션은 부드러운 보간이 아니라 **뚝뚝 끊기는 프레임 교체**라 오히려 쉬움:
- idle 스프라이트: PNG 2~3장을 `Timer.periodic(300ms)`로 교체하는 `StatefulWidget` 하나 (백엔드 폴링 루프와 같은 멘탈 모델)
- 버튼 눌림/흔들림: 내장 `AnimatedContainer`/`AnimatedScale`에 duration 짧게(100~200ms)
- 픽셀 유지: `Image.asset(..., filterQuality: FilterQuality.none)` 필수
- Flame(게임엔진)은 오버킬 — 목장에 양 수십 마리 돌아다니는 수준이 되면 그때 검토

### 3.5 픽셀 동물 카드 수집 시스템 (핵심 결정 A 구현, 백엔드 스키마 변경 수반)

**획득 (트리거)**
- [ ] **섹터(섹션) 클리어 시 카드 팩 1개 지급** — 자잘한 구절마다 주면 카드 인플레이션이 심하므로 섹터 단위 권장. (구절 1개 green은 조각/코인 등 소액 보상으로 별도 검토)
- [ ] 팩 개봉(언박싱) 연출 — 픽셀 카드가 뒤집히며 등장, 희귀도 높을수록 화려한 반짝임
- [ ] 카드 획득은 순수 보상 — 안 해도 벌 없음("가볍게" 원칙), 암송이 획득 트리거

**카드 로스터 & 희귀도**
- [ ] 고정 로스터 30~50종 확정 (성경 동물 + 일반/재미 동물). 우선 성경 동물부터:
  - 성경 동물: 비둘기(제작됨), 양, 까마귀, 참새, 독수리, 수탉, 사자, 뱀, 물고기, 메추라기 등
  - 재미/희귀: 코끼리, 공룡 등 (전설 등급, "왜 성경 앱에 이게?"의 유쾌함)
- [ ] 희귀도 4단계 + 픽셀 프레임 구분:

  | 등급 | 프레임 | 예시 |
  |---|---|---|
  | 일반 | 구리 | 참새, 양 |
  | 레어 | 은색 | 비둘기, 까마귀 |
  | 에픽 | 금색 홀로 | 독수리, 사자 |
  | 전설 | 무지개 반짝임 | 공룡, 코끼리 |

- [ ] 각 성경 동물 카드에 성경 근거 한 줄(예: 비둘기=노아의 방주·성령, 독수리=이사야 40:31) — 카드 뒷면/상세에 표기해 학습 연계

**프로토타입 검증 완료 (2026-07-19)**: 4등급 카드 프레임(구리/은/금 홀로/무지개)을 HTML+CSS로 시제작해 다크모드 배경에서 확인. 에픽은 대각선 홀로 반짝임(`background-position` 이동 애니메이션), 전설은 무지개 그라데이션 테두리(`hue-rotate` 회전) + 동일한 대각선 홀로를 이중 적용. 하드섀도 카드 프레임에 등급 프레임만으로 희귀도가 즉시 구분됨을 확인.
- **Flutter 구현 가능성 확인**: 전부 1:1 대응 가능 — 하드섀도(`BoxShadow`), 무지개 테두리(`gradient` `BoxDecoration`), 홀로 반짝임(`ShaderMask`+`LinearGradient`+`AnimationController`, 흔한 shimmer 패턴), 픽셀 유지(`FilterQuality.none`). 게임엔진(Flame) 불필요.
- **성능 방침**: 무지개 hue-rotate를 카드 여러 장에 동시 재생하면 저사양 기기 부담 → 도감 목록은 정적 프레임, **팩 개봉·카드 상세 화면에서만 풀 애니메이션** 재생으로 최적화(도감 화면 설계에도 반영).

**도감 (컬렉션) — 화면 설계**

진입: 하단 탭바 또는 대시보드에 "도감" 진입점 (5단계 창 문법과 통일 — 각진 박스+선택 `▶`).

- [ ] **상단 진행 요약 바**: "28/40 수집" 카운터 + 픽셀 진행 게이지(`.memorize-progress` 세그먼트 스타일 재사용). 등급별 소계도("전설 1/3") 함께
- [ ] **그리드 바인더**: 카드를 3~4열 그리드로 나열. 포켓몬 도감처럼 **번호 순 고정 슬롯**(빈 칸도 자리를 지킴)이라 "몇 번이 비었는지" 한눈에 보임
  - 획득: 컬러 카드 썸네일 + 등급 프레임 + 보유 별(★★☆)
  - 미획득: **검은 실루엣**(스프라이트를 단색으로) + `???` — 다음에 뭘 모을지 궁금하게(수집 동기)
  - 잠김 힌트: 미획득 카드 탭 시 획득 조건 한 줄("○○ 코스 완주 시 등장" 정도, 스포일러는 최소화)
- [ ] **필터/정렬**: 등급별 필터(전체/일반/레어/에픽/전설), 정렬(도감 번호순 / 최근 획득순 / 미획득 우선). 초기엔 등급 필터만이라도
- [ ] **카드 상세(탭 시 확대)**: 카드가 크게 뒤집히며 등장(개봉 연출 재사용) + 성경 근거 전문 + 보유 수량/별 등급 + 획득한 날짜. 홀로 애니메이션은 이 상세 화면에서만 풀로 재생(도감 목록에서는 정적 프레임 — 저사양 기기 성능 최적화)
- [ ] **신규 획득 뱃지**: 아직 상세를 안 열어본 새 카드에 `NEW` 픽셀 뱃지 → 도감 재방문·확인 동기

**향후 확장(비범위, 아이디어만 기록)**
- 시리즈/세트 개념: "구약 동물 세트", "예수님의 비유 세트" 등 테마별 묶음. 세트 완성 시 보너스 카드/칭호
- 도감 공유: 수집 현황을 픽셀 카드 이미지로 SNS 공유(기존 share-card 인프라 연계)
- 카드 배경 변주: 같은 동물이라도 코스별로 다른 배경(사막/바다/성전)을 입혀 수집 깊이 추가

**중복 처리 = 복습 동기**
- [ ] 같은 카드 중복 획득 시 별 등급 업(★→★★→★★★) 또는 코인 전환 — 이미 깬 섹터를 다시 완벽하게 깨서 카드를 업그레이드하는 자연스러운 복습 루프. **억지 푸시 알림 없이 재방문 이유를 만든다**(다마고치의 알림 강제를 대체하는 핵심 장치).

**필요 백엔드 작업**: 사용자별 보유 카드/수량/등급 저장 테이블, 카드 마스터 데이터(로스터·희귀도·성경 근거) 테이블. 3번(데일리 루프)·섹터 클리어 판정과 연동되므로 스키마를 함께 설계하는 게 효율적.

**완료 기준**: 섹터를 클리어하면 카드 팩이 열리고, 도감에서 수집 현황을 볼 수 있고, 중복 시 업그레이드된다. 모든 획득은 무보상 벌 없이 "가볍게" 유지.

**비범위(이번 컨셉에서 명시적으로 제외)**: 밥주기·진화 게이지·강제 알림 등 다마고치식 의무 요소 일체. 유료 팩 구매(가챠 과금)는 별도 [구독 계획](subscription-implementation-plan.md)에서 다룸 — 초기엔 암송으로만 획득.

## 진행 순서

1(팔레트) → 2(마스코트) → 3(데일리 루프) → 3.5(카드 수집) → 4(완료 화면) → 5(창 문법·헤더·버그) → 6(정리) → 7(ASO) → 8(앱/웹 기능 구별)

1번은 CSS 변수만으로 즉시 체감, 먼저 처리. 2번은 에셋 제작이 필요해 가장 오래 걸리지만 임팩트 최대. 3번은 마스코트가 있어야 보상 연출이 완성되므로 2번 뒤. **3.5번(카드 수집)은 3번의 섹터 클리어 판정과 백엔드 스키마를 함께 설계해야 마이그레이션이 한 번으로 끝남.** 단, 3번(데일리 루프)이 실제로 재방문율을 올리는지 먼저 가볍게 검증한 뒤 3.5의 카드 로스터/일러스트 대량 제작에 착수하는 게 안전 — 3번은 프론트 위주로 작게 낼 수 있으니 검증 후 3.5로. 5번의 로그아웃 배지 버그는 우선순위 높여 먼저 고쳐도 무방(작은 수정). 6번은 언제든 끼워 넣을 수 있는 정리 작업. 7(ASO)은 비주얼이 어느 정도 정리된 뒤 스크린샷을 찍어야 하므로 4~6 이후가 자연스러움. 8(앱/웹 기능 구별)은 구독 구현 착수 전에 선행되어야 함 — [구독 구현 계획](subscription-implementation-plan.md)의 전제 조건.
